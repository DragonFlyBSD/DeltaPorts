# BSD type names invisible under strict POSIX flags

## Pattern

The compile fails because a BSD legacy type name (`u_short`, `u_int`,
`caddr_t`, `daddr_t`, `quad_t`, `u_char`, `u_long`, `ushort`, `uint`)
is reported as unknown. The error originates inside a DragonFly
system header that uses the type in a struct field:

```
/usr/include/sys/event.h:67:2: error: unknown type name 'u_short'
  u_short  flags;
  ^~~~~~~
```

The build command (visible in `dsynth_log` output) carries one or
both of:

- `-D_POSIX_C_SOURCE=...` (any value)
- `-D_XOPEN_SOURCE=...` (any value)

The offending header is a BSD system header (`<sys/event.h>`,
`<sys/socket.h>` in some configurations, `<sys/select.h>`, etc.) —
NOT user code under WRKSRC. The user code is including the system
header at the top of a unit and the system header itself is where
compilation halts.

## Cause

DragonFly's `<sys/cdefs.h>` translates user-facing feature-test
macros into libc-internal visibility flags:

- `_POSIX_C_SOURCE` → `__POSIX_VISIBLE`
- `_XOPEN_SOURCE` → `__XSI_VISIBLE`
- `_BSD_SOURCE` (or absence of any restriction) → `__BSD_VISIBLE`

When the user requests strict POSIX visibility (`_POSIX_C_SOURCE=200809L`
or similar), `__BSD_VISIBLE` is cleared. `<sys/types.h>` then does
NOT declare the BSD legacy type aliases (`u_short`, `u_int`, etc.).
Any subsequent BSD system header that uses those types in its public
struct definitions fails to compile, because the type identifiers
the header references are no longer in scope.

The fix is to either drop the restricting flag or to opt back into
BSD visibility via the user-facing flag. **Never** reach for the
internal `__BSD_VISIBLE` — see Anti-pattern below.

## Fix — ranked best-to-worst

### Option 1 (preferred): remove the restricting macros

Find where `-D_POSIX_C_SOURCE` and/or `-D_XOPEN_SOURCE` are set.
Usually one of:

- The upstream `configure` script sets them unconditionally (look
  for `-D_POSIX_C_SOURCE` in the autogenerated Makefile or in
  `configure.ac`).
- The upstream `Makefile` (hand-rolled, not autoconf) hardcodes
  them in `CFLAGS`.
- A `make/exports.sh` or similar shell stanza in the project
  exports them as environment.

The strict-POSIX request is usually a portability shim the project
inherited from a different platform. DragonFly doesn't need it.
Drop the flag on DragonFly:

- Autoconf project: emit a `dragonfly/patch-Makefile.in` (or whatever
  the generated Makefile is) via `add_patch` that removes the
  `-D_POSIX_C_SOURCE=...` token from the relevant `CFLAGS` line.
- Hand-rolled Makefile: same, but patch the source `Makefile` directly.
- Configure-script generated: same — patch the post-configure Makefile,
  not `configure.ac` (the agent's overlay applies AFTER configure runs).

This is the cleanest fix because it removes the false constraint
rather than working around it.

### Option 2: add `-D_BSD_SOURCE` to opt back in

If removing the POSIX flag would be invasive (the upstream gates
many feature checks on it), add `_BSD_SOURCE` alongside. This is
the canonical user-facing escape hatch — `<sys/cdefs.h>` recognizes
it and re-enables BSD-namespace declarations without disturbing
the existing POSIX-conformance code paths.

```dops
mk add CFLAGS "-D_BSD_SOURCE"
```

`_BSD_SOURCE` is the API. `__BSD_VISIBLE` is the implementation.

### Option 3 (last resort): include `<sys/types.h>` explicitly

If the failing BSD system header doesn't pull in `<sys/types.h>`
itself, adding the include via a source patch CAN help on some
platforms — `<sys/types.h>` defines the BSD aliases regardless of
visibility on certain libcs.

On DragonFly's libc, this is **not reliable**: `<sys/types.h>`
itself gates the BSD aliases on `__BSD_VISIBLE`, so adding the
include doesn't change visibility — the aliases stay hidden under
strict POSIX. Treat this option as a fallback only when Options 1
and 2 are infeasible.

## Anti-pattern: do NOT define `__BSD_VISIBLE` (or any `__`-prefixed macro)

`__BSD_VISIBLE`, `__POSIX_VISIBLE`, `__XSI_VISIBLE`, `__ISO_C_VISIBLE`
are *libc implementation details*. They live in the C-standard
reserved namespace (§7.1.3: identifiers beginning with two underscores
or with an underscore followed by an uppercase letter are reserved
for the implementation). They are SET BY `<sys/cdefs.h>` from the
user-facing flags; they are not the API.

Defining them from CFLAGS:

- **happens to work** on DragonFly's libc because that's the
  internal name DragonFly uses,
- **silently breaks** on any libc whose internal name differs
  (glibc uses `__USE_BSD` for the analogous concept; musl uses
  yet another scheme),
- **invokes undefined behavior** per the C standard,
- **obscures the cause** — a porter reading the patch can't tell
  from `-D__BSD_VISIBLE` alone that the actual issue is the
  upstream's `-D_POSIX_C_SOURCE` four levels up the CFLAGS chain,
- **can collide with future libc changes** if the implementation
  renames its internal macros.

### If a previous fix already shipped `-D__BSD_VISIBLE`

The fix compiles and the test bot is green, but it's wrong by the
criteria above. On the next iteration (operator-driven retry,
upstream version bump, etc.):

1. Read `overlay.dops` — find the `mk add CFLAGS "-D__BSD_VISIBLE"`
   (or equivalent `mk set`) line.
2. Emit `change_makefile op=remove` to drop the `-D__BSD_VISIBLE`
   token from CFLAGS (or `op=unset` if it's a standalone assignment).
3. Apply Option 1, 2, or 3 from above as the replacement.
4. Re-run the build to confirm the cleaner fix still passes.

## Examples

- `devel/skalibs` (2026-06-01, commit `a8d053f050f`): upstream
  `make/exports.sh` adds `-D_POSIX_C_SOURCE=200809L -D_XOPEN_SOURCE=700`
  for portability. First-pass agent emitted `mk add CFLAGS
  "-D__BSD_VISIBLE"`. Compiled and shipped, but Option 1 (patch
  `make/exports.sh` to drop the POSIX flags on DragonFly) would
  have been the right shape. Worth re-fixing the next time the
  port needs touching.
